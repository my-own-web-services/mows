# VideoViewer

Fully-featured video player for the file-viewer dispatch. Built on
[`shaka-player`](https://github.com/shaka-project/shaka-player) (v5) so the
same component plays:

- progressive standard formats — `video/mp4`, `video/webm`, `video/ogg`, …
- adaptive streaming — DASH manifests (`application/dash+xml`) and HLS
  playlists (`application/vnd.apple.mpegurl`, `application/x-mpegurl`, …)

Shaka loads the bundle through `MediaSource Extensions` for everything,
falling back to native `<video src>` only when `shaka.Player.isBrowserSupported()`
returns false (rare — old iOS Safari, no-MSE environments).

This component is **heavy** — shaka-player is ~256 kB gzipped — so
`FileViewer` imports it via `React.lazy()`. If you render `VideoViewer`
directly, code-split it the same way unless your route always wants video
playback eagerly.

## Backend requirements

For seek + adaptive streaming to work end-to-end the file server **must**:

- support HTTP Range requests (return `206 Partial Content` with
  `Content-Range` + `Accept-Ranges: bytes`),
- include `Content-Range`, `Accept-Ranges`, `Content-Length`, and
  `Content-Type` in CORS `Access-Control-Expose-Headers` so Shaka can read
  them from JS.

Without the exposed headers, progressive playback works but DASH/HLS
manifests fail to load segments.

## Controls

The control bar is built entirely from `lib/components/ui/` primitives
(`Button`, `Slider`, `DropdownMenu`, `Tooltip`) — no raw HTML controls. It
auto-hides 2.5 s after the last pointer movement during playback and
re-appears on the next move, click, or while paused.

| Affordance         | Primitive                     |
| ------------------ | ----------------------------- |
| Play / pause       | `Button` (icon)               |
| Seek               | `Slider` (with preview tooltip) |
| Volume + mute      | `Slider` + `Button`           |
| Quality selector   | `DropdownMenu` (shows ABR's active pick when on Auto) |
| Captions selector  | `DropdownMenu` (only when text tracks exist) |
| Playback rate      | `Popover` slider 0.25× – 3× plus preset chips |
| Picture-in-picture | `Button` (Tooltip)            |
| Fullscreen         | `Button` (Tooltip)            |

### Seek bar — drag, commit, preview

The seek bar is split into three distinct visual states so the user is never
misled about what the player is doing.

| State               | What the slider shows                              |
| ------------------- | -------------------------------------------------- |
| Idle                | `currentTime` of the underlying `<video>` element. |
| Dragging            | The pointer's position. The main `<video>` is **also** scrubbed live (throttled to one seek per animation frame) so the player canvas shows the new frame as the cursor moves. |
| Released, buffering | Pinned to the release position until the video's `currentTime` arrives within 0.6 s of the seek target (5 s watchdog fallback). The slider does **not** snap back to the pre-seek position while the player is still loading the new range. |

Above the seek bar a floating tooltip surfaces every hovered / dragged
position:

- The **timestamp pill** is always rendered as soon as the pointer enters
  the bar (or a drag begins) — even for sources that can't produce a frame.
- The **thumbnail tile** appears whenever the consumer is wired up to
  produce previews (image-track manifest or `video/*` source for the
  frame-grab fallback). While the frame for the current time is still
  being decoded, the tile shows an empty placeholder with a spinner
  instead of holding on to the previously-decoded tile — the user never
  sees a thumbnail that doesn't match where the cursor is.
- Thumbnail requests **coalesce**: only the most-recent pending time
  actually drives a seek on the hidden grabber video. Intermediate
  positions the cursor passed over are skipped, so drag-scrub stays
  close to live instead of crawling through every queued frame.

### Preview-thumbnail sources

In priority order, the `VideoViewer` looks up the tile via:

1. **Native image tracks** — `shaka.player.getImageTracks()` is non-empty,
   meaning the manifest exposes a sprite-sheet thumbnail adaptation set
   (e.g. DASH `AdaptationSet contentType="image"` with
   `EssentialProperty schemeIdUri="http://dashif.org/guidelines/thumbnail_tile"`).
   These are typically pre-generated by the packager (Shaka Packager, AWS
   MediaConvert, Bento4, …) at ingest time.
2. **Frame-grab fallback** — for plain `video/*` sources without image
   tracks. `VideoFrameGrabber` mounts a hidden, off-screen `<video>` +
   `<canvas>` pair, seeks the hidden video to the requested time, and
   draws the current frame into a 160 × 90 box. The blob URLs are LRU-
   cached (32 entries) so repeat hovers don't redo the work.
   - Requires CORS — the source must return `Access-Control-Allow-Origin`
     so the canvas isn't tainted. Tainted canvases throw `SecurityError`
     in `drawImage` / `toBlob`, in which case the SeekPreview gracefully
     drops back to timestamp-only.
   - Requires HTTP Range support on the file URL (already needed for
     normal playback seek).
   - DASH / HLS sources **cannot** use the frame grabber — a plain
     `<video src>` can't decode a manifest. Adaptive streams without an
     image track render timestamp-only previews.
3. **Timestamp-only** — when neither path is available.

The hidden grabber `<video>` lives inside a `1 × 1 px`, `position: fixed`,
`opacity: 0` host appended to `document.body`. Modern Chromium throttles
media on **detached** elements (the `seeked` event may never fire), so the
host element is required even though it's never visually presented.

### Chapters

Pass a `chapters: ReadonlyArray<Chapter>` prop and the seek bar grows
YouTube-style tick marks at every chapter boundary. The hover/drag tooltip
surfaces the matching chapter title above the timestamp, so the user sees
both *where in the clip* and *which chapter* at a glance.

```ts
interface Chapter {
    readonly id: string;            // React key only
    readonly title: string;          // shown in the tooltip
    readonly startTime: number;      // seconds
    readonly endTime?: number;       // optional; defaults to the next chapter's start
}
```

Chapters are sorted by `startTime` before rendering, so the consumer
doesn't have to. A chapter starting at `0` skips its tick (the bar's left
edge already marks the boundary). Sources that don't ship chapters omit
the prop entirely — the tooltip falls back to timestamp-only.

### Keyboard shortcuts

The wrapper is `tabIndex={0}`; once focused these bindings fire (none claim
modifier-combined keys, so browser shortcuts keep working):

| Keys             | Action                          |
| ---------------- | ------------------------------- |
| `Space`, `k`     | Toggle play / pause             |
| `←` / `→`        | Seek ±5 s                       |
| `↑` / `↓`        | Volume ±10 %                    |
| `m`              | Toggle mute                     |
| `f`              | Toggle fullscreen               |
| `p`              | Toggle picture-in-picture       |
| `c`              | Toggle captions                 |
| `<` / `>`        | Slower / faster playback rate   |

## Lifecycle

- `componentDidMount` checks `shaka.Player.isBrowserSupported()`. If false,
  switches to the native `<video controls>` fallback path and skips Shaka.
  Otherwise constructs one `shaka.Player`, attaches it to the `<video>`
  ref, subscribes Shaka events (`trackschanged`, `adaptation`, `buffering`,
  `error`) and DOM events (`timeupdate`, `loadedmetadata`, `play`, `pause`,
  `volumechange`, `ratechange`, `ended`, `enterpictureinpicture`,
  `leavepictureinpicture`), then `player.load(src)`.
- `componentDidUpdate` re-`load`s the existing player on `src` change — the
  player instance is **not** torn down between sources.
- `componentWillUnmount` flips a `cancelled` flag (StrictMode-safe) and
  fires `player.destroy()`.

## Props

| Prop        | Type                          | Notes                                                                 |
| ----------- | ----------------------------- | --------------------------------------------------------------------- |
| `src`       | `string` (required)           | Resolved URL: a file URL or a manifest URL (DASH/HLS).                |
| `mimeType`  | `string` (required)           | Drives behaviour & rendered in the aria-label.                        |
| `name`      | `string`                      | Surfaced as the wrapper's `aria-label`.                               |

| `poster`    | `string`                      | Forwarded to `<video poster>`.                                        |

## States

- **Loading** — `Loader2` spinner overlay while shaka is buffering / loading
  metadata.
- **Playing / paused** — control bar visible while paused, auto-hides during
  playback after 2.5 s of pointer idle.
- **Error** — translated error overlay with a Retry button that re-loads
  the same `src`. Triggered by either the Shaka `error` event or a load
  rejection.
- **Native fallback** — `shaka.Player.isBrowserSupported()` returned false.
  The custom control bar is hidden and `<video controls>` shows native
  chrome instead.

## Notes

- Shaka v5 removed `setTextTrackVisibility`. Captions visibility is
  expressed by selecting (`selectTextTrack(track)`) or unselecting
  (`selectTextTrack(null)`) a text track.
- ABR auto-selection is toggled via `player.configure({abr: {enabled}})`;
  picking a specific quality also disables ABR until the user picks
  **Auto** again.
- The video element is `playsInline` so iOS doesn't hijack into native
  fullscreen the moment playback starts — fullscreen happens via the
  wrapper's `requestFullscreen()` so our custom controls stay on top.
