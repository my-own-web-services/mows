### Security

✅ **SECURITY-1 / SECURITY-2** — `utils/mows-cli/src/agents/commands.rs:583-630, 636-695` — private-key writes refactored to two helpers (`ssh_scratch_dir`, `write_private_key`). Keys are now written to `$XDG_RUNTIME_DIR/mows-cli/` (per-user 0700 tmpfs on systemd hosts) or `$TMPDIR/mows-cli-$UID/` (chmod 0700) — never the world-traversable `/tmp` root. `create_new(true)` makes the write atomic; an attacker cannot pre-create a file at the target path to inherit permissive perms.

⁉️ **SECURITY-3** — `utils/mows-cli/src/agents/bootstrap.rs:371-374` — claim: $HOME bind-mount + NET_ADMIN/SYS_ADMIN = container escape. Verified: the supervisor runs as the user, acting on the user's behalf; the bind-mount is documented (lines 371-372) and required because users pass arbitrary CWDs as VM workspaces. Threat model permits it — same trust boundary as `mows` itself.

⁉️ **SECURITY-4** — `utils/mows-cli/src/self_update/update.rs:867-883` — claim: `bash builder_path` without canonicalize is path-traversal vulnerable. Verified: the path is found by walking up from CWD looking for `utils/mows-vm-supervisor/image-builder/build.sh`. An attacker would need write access to the user's repo paths — at which point they already have arbitrary execution. Canonicalising does not change this.
